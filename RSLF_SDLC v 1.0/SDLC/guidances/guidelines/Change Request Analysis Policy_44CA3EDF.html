Guideline: Change Request Analysis Policy
The purpose of the Change Request Analysis and Resolution policy is to provide standard process and approach used to report, analyze, resolve and make decision on change requests that may occur during the software project.

Main Description

Overview

This policy provides a standard approach to identifying the potential consequences of a change, or estimating what needs to be modified to accomplish a change and accordingly make decision on that change whether by acceptance or rejecting the change request taking into consideration that change request is submitted only for artefacts and work products that have been baselined otherwise no need for submitting change requests.

 

Change Request Approval Process

This section outlines the process for requesting and analyzing changes to an application during the software development cycle. The goal of this process is to produce a more robust approach that supports the decision making on requested changes. The process is illustrated as follows;

 

 

Study Requested Changes Impact

This task ensures that due consideration is given to the impact of change on the project and that change requests are studied and analyzed within the project in a consistent manner.

 

Make Decision on Change Request

This task is designed to make decision on a change request by accepting or rejecting based on the given criteria and impact analysis.

 

Change and Deliver Configuration Items

This task ensures that changes have been occurred on the required work products to be changed as stated in the change request.

 

Change Request Evaluation Attributes

The standard evaluation attributes describing scope, size and effort for a change request and helping project management team make decision on this change request are illustrated as follows:

The Impact of the change requested needs to be specified for the following parameters.

1.       Technical – Has an impact of the technical artefacts of the project. May impact multiple artefacts and deliverables through the entire lifecycle of the project

2.       Effort – Has an impact on the project effort, this may or may not impact the project schedule.

3.       Time Duration – This has a direct impact on the schedule of the project. There is a high chance that the effort for the project is also impacted.

4.       Work Procedures – This is particular applicable if there needs to a change request that needs to be raised for processes defined in the SCHT  SDLC. This is used if the path followed is not via the NPI process management approach.

5.       Financial – The Financial impact on the project parameters.

6.       Contractual

7.       Impact on other projects, which may be linked to the current project.

 

Decision on Change Request

Judgment or decision making to accept or reject the change request is provided based upon expertise in the software application change type as appropriate for the activity being performed. Such expertise may be provided by any group or person in the project with specialized, knowledge, skills and experience. The Project manager may use the expert judgement as a basis to determine the overall impact on the project, depending on the current state of execution of the project.

Specific approvals from the management are required as a part of providing the approval for the change request. 

As a result of the change request acceptance, the team members are assigned into the software project schedule for completing, changing and delivering Configuration Items required to be changed.

 

If the change in schedule less than or equal 5% in one phase, only need to take approval from PM
