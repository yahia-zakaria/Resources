Main Description

Overview

The Solution Architecture Definition (SAD) document describes the subsystems and components of the solution by presenting a number of architectural views. Each view shows a different aspect of the application to address different concerns, below are the main categories should be covered:


1.    Introduction

[The introduction should provide an overview of the entire Solution Architecture Document. It should include the purpose, scope, definitions, acronyms, abbreviations, references]

1.1  Purpose

[This section defines the purpose of the Solution Architecture Document, in the overall project documentation, and briefly describes the structure of the document. The specific audiences for the document should be identified, with an indication of how they are expected to use the document.]

1.2  Definitions, Acronyms, and Abbreviations

[This subsection should provide the definitions of all terms, acronyms, and abbreviations required to properly interpret the Solution Architecture Document.]

1.3  References

[This subsection should provide a complete list of all documents referenced elsewhere in the Software Architecture Document.  Each document should be identified by title, report number (if applicable), date, and publishing organization.  Specify the sources from which the references can be obtained. This information may be provided by reference to an appendix or to another document.]


2.   Architectural Goals and Constraints

[This section describes the software requirements and objectives that have some significant impact on the architecture, for example, safety, security, privacy, portability, distribution, and reuse. It also captures the special constraints that may apply: design and implementation strategy, development tools, team structure, schedule, legacy code, and so on.]


 

3.   Use-Cases Realizations

[This section illustrates how the software actually works by giving a few selected use-case (or scenario) realizations, and explains how the various design model elements contribute to their functionality.]


4.   Logical View

[This section describes the architecturally significant parts of the design model, such as its decomposition into subsystems and packages. And for each significant package, its decomposition into classes and class utilities. You should introduce architecturally significant classes and describe their responsibilities, as well as a few very important relationships, operations, and attributes.]


5.   Process View

[This section describes what the system is doing from high-level, and it also provides you to show how the smaller components, tasks, threads fit together and involved in the system execution.]


6.   Data View

 [This section describes system data model and its organization in terms of the tables, views, indexes, triggers and stored procedures used. This section is optional if there is little or no persistent data, or the translation between the Design Model and the Data Model is trivial.]


7.   Deployment View

[This section describes one or more physical network (hardware) configurations on which the solution is deployed and run. It is a view of the Deployment Model. At a minimum for each configuration it should indicate the physical nodes (computers, CPUs) that execute the software, and their interconnections (bus, LAN, point-to-point, and so on.) Also include a mapping of the processes of the Process View onto the physical nodes.]


8.   Size and Performance

[This section describes the major dimensioning characteristics of the solution that impact the architecture, as well as the target performance constraints.]


9.   Issues and concerns

[This section describes any issues or concerns.]


The Solution architecture policy should consider the below principles: 


Principle #

Description

SCP0001

Decisions are made with the intention of achieving maximum benefit for client

SCP0002 

Applications are developed for use across the enterprise in preference to similar or duplicative applications which are provided to a particular department

SCP0003

Each data element has an Information System Owner accountable for data & quality

SCP0004

Data is defined consistently across the enterprise and the definitions are understandable and available to all users

SCP0005

Information is protected from unauthorized use and disclosure

SCP0006

Applications are independent of specific technology choices and therefore can operate on a variety of technology Platforms

SCP0007

Applications are easy to use and the underlying technology is transparent to users so they can concentrate on tasks at hand.

SCP0008

Changes to applications and technology are only made in response to business needs

SCP0009

Technological diversity is controlled to minimize the non-trivial cost of maintaining expertise in and connectivity between multiple processing environments

SCP0010

Software and hardware conform to defined standards that promote interoperability for data, applications and technology

SCP0011

Middleware should be used to decouple applications from specific software solutions.

SCP0012

The application should be built by the latest technologies and ready to adopt a future version of that technology.

SCP0013

The Application should support scalability to accommodate additional utilization, users, transactions, and/or additional local/remote sites.

SCP0014

The application should be able to extend its view to support Mobile

SCP0015

The application should have an open interface to support connectivity with other systems once needed.

SCP0016

Mobile applications must support Android and IOS systems.

SCP0017

Applications must support data auditing and tracking.

SCP0018

The solution should fully support localized languages including native full Arabic and English support as well as Unicode.

SCP0019

Software architecture must support software-based load balancing for scalability and performance.

SCP0020

Software architecture must support trace logging, error notification and resolution and exception handling.

SCP0021

The application must support three-tier architecture implementation

SCP0022

Data Reporting & Statistics requires to integrate with existing platform to provide data where necessary to support the requirements for the solution.

 

Solution Architecture Review 


The primary purpose of solution reviews is to identify anomalies in the review item, so that these can be corrected. This policy provides minimum acceptable requirements for solution architecture.

 

A systematic evaluation of a solution architecture work product by a team of qualified personnel that examines the verity and suitability of the work product for its intended use and identifies discrepancies from specifications and standards based on standard checklists :
