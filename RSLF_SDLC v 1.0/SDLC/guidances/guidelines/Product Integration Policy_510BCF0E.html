Product Integration Strategy

Integration strategy start from development, where each develop have separated development  for development and writing unit testing and run smoke test before developer deliver the component to Team Leader by checking in code on TFS.

Team Leader collect the components from TFS and deploy them on his development machine and check the integrity between the different components and solve any issues related to it with the developers, after that he run smoking test and unit testing on service layer if available.

Then team leader merge code from DEV to Main in TFS and build then deploy on local test environment and generate release note and deliver it to Tester.

Tester review the release note then run testing according the test cases in TFS and add bugs and assign them to team leader in TFS, team leader review the bugs and reassign them to developer who is responsible about the component, after developer fix on MIAN code base he assign the bug to team leader to do smoke testing , then team leader  assign again to tester for testing and closing by tester.

Team Leader after the bugs closed on TFS and the release  is ready to be deployed on the production environment, he create or update if the delivery will be on multiple releases  deployment document to be deliver to Deployment Engineer, he  also Merge Code from MAIN  to DEV code base  to update the Dev with fixes that happen on MAIN, then he Merge from MAIN to Production.

 Deployment Engineer receive the Packages on Media (Flash, HD, CD) and the deployment document and schedule meeting on the client side to check the readiness of production environment according  to design  document , after he verify that is it according the design document , he deploy the release and make smoking test on the product , he report any bug to team leader by adding it as bug on TFS.

Client test the product under supervision of the deployment engineer or team leader to get User Acceptance testing  and report the result to the project manager.

If the bugs is repeated Team leader will add them with how to resolve them to Knowledge base on TFS to be supporting to other projects, by studying them we can update process and document and development practices .

Integration Procedure and Criteria

 The Product is divided into collection of components to simplify the implementation.

Developer must run smoking testing on his environment including (data input Validation, different Users types usages and security privileges, UI is applied correctly).

Developer write unit testing for all functionality of the component that expose through service layer and run this unit testing  before he deliver the code to team leader .

Team leader run smoking  and unit testing on his environment for each component he receive from the developers.

Team Leader run Integration testing after he integrate the comments on single product.

Smoking and Integration testing must passed and their no tolerance or the components will be excluded from the release note.

Tester Categories bugs on three types (can refer to Test cases guidlines document)

              1- Functionality:

This mean that the component is missing main functionality and not delivered to client

2- UI:

              This mean that related to how the graphic design and user experience and if it not stopping the functionality the delivery of the release can be done , but not on final iteration of delivery it must be solved before it.

 3- Compatibility 

4- Checking across multiple browsers
