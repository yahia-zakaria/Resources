Guideline: Work Breakdown Structure Policy
The purpose of the Work Breakdown Structure (WBS) Policy is to provide concepts of SDLC WBS and standard application of the WBS as a project management tool.

Main Description

Overview

The intent is to establish a consistent development of the WBS as a project management artifact and as a result improve in a unified manner the planning and controlling of software projects.

 

WBS Definition

Project is made more manageable by breaking it down into individual components that together are known as a Work Breakdown Structure or WBS. Such a structure defines unique work elements that can be arranged and completed in the order defined by the network diagram: sequentially, in parallel, or in the specific order necessary to accomplish the software project outcomes. It facilitates other software project management processes such as estimating, scheduling, resource allocation, risk analysis, and measurement and control of the project.

 

The WBS represents a clear description of the software project’s deliverables and scope—the ‘‘what’’ of the project. It is not a description or schedule that defines how or when the deliverables will be produced, but rather is specifically limited to describing and detailing the software project’s outcome or scope.

 

(PMO) defines the WBS as follows:

‘‘The WBS organizes and defines the total scope of the software project. The WBS subdivides the software project work into smaller, more manageable pieces of work, with each descending level of the WBS representing an increasingly detailed definition of the project work. The planned work contained in the lowest level WBS components, can be scheduled, cost estimated, monitored, and controlled.’’

 

WBS Levels

The WBS includes all work to be done by the project leaders, stakeholders, and both internal and external participants, such as team members and vendors. The WBS provides a clear statement of the objectives and deliverables of the work to be performed.

 

The depth of a WBS is dependent upon the size and complexity of the software project and the level of detail needed to plan and manage it. Most software work breakdown structures consist of a multi-level hierarchy describing the entire scope to be accomplished by the performing organization; however, the specific number of levels should be appropriate for effectively managing the project in question.

 

WBS Consideration

In developing a WBS, the following basic tenets should be considered:

 

1.       All WBS elements are compatible with the WBS model.

2.       All WBS elements are compatible with organizational structures and roles.

3.      The WBS should include the standard phases, iterations, activities, tasks, milestones and checkpoints.

4.      Each WBS element represents a single tangible or intangible work product.

5.      Work products include both final and interim work products that are required to create the final results.

6.      Each requirement work item is represented in terms of task or more and included in the WBS in order to be tracked and monitored.

7.      The work products are logically decomposed to the level that represents how they will be produced and managed.

8.      Work products included in the WBS are unique and distinct.

9.      All significant reporting mechanisms and scheduled administration, such as review meetings, monthly reports and status meetings are included and identified in the WBS.

10.   Accountability for each task must be assigned to a single project team member from internal or vendor related. If this is not possible, then reconsider whether or not the task can be further decomposed.

11.    A coding scheme is used for WBS elements that clearly represent the hierarchical structure when viewed in text format.

12.    There is no need to have all work packages at the same level.
