Main Description

Overview

The purpose of the Software Configuration Management policy is to ensure that standardized procedures and methods are used to (1) identify and document the functional and physical characteristics of a configuration item, (2) control changes to those characteristics, (3) record and report change processing and implementation status, and (4) verify compliance with specified requirements.

 

General Consideration

 

·         A documented configuration management plan must be established and published.

·         Affected groups and individuals must be informed of the status and content of software baselines.

·         Configuration management library system and repository must be available.

·         Changes to identified work products are controlled and managed.

·         The Configuration management for identified work products (both external and internal) must be implemented.

·         Documents and project artifacts must be reviewed as per the review policy.

·         The change management process for all identified work products (both external and internal) must be implemented as per the documented change management process.

·         Approval of senior management for all deviations from the documented policy must be identified.

·         Measurement reports to determine the status of configuration management activities must be used and reflected in the periodical status report.

·         The Project Manager & Configuration Manager are responsible for implementing this policy.

 

 

Configuration Management

 

Configuration Management (CM) is a set of activities to establish and maintain the integrity of the work products of the software projects. CM involves identifying the configuration of the software work products (containing identified items and their descriptions) at given point in time, systematically controlling changes to the configuration, and maintaining the integrity and traceability of the configuration throughout the software life cycle. CM identifies, organizes, controls, audits and reports modifications that invariably occur while software work product is being developed and even after it has been released to users. CM ensures that system requirements can be traced to the final product.

 

In early stages of software engineering process, CI’s such as Business Analysis , Software Project Charter, Project Plan, Software Test Plan, Software Configuration Management Plan are developed.

As the software engineering process progresses more CI’s, Software Architecture, Source Code, Test Cases, and Documentation are developed.

The Configuration Management process involves the following activities:

 

·         Identifying the configuration of selected work products that compose baselines at given points in time.

·         Controlling changes to configuration items.

·         Building or providing specifications to build work products from the configuration management system.

·         Maintaining the integrity of baselines.

·         Providing accurate status and current configuration data to developers, end users, and customers.

Configuration Management Tasks

 

The CM tasks required to be performed in a project are:

 

1)      Establish Configuration Management Plan by Configuration Manager.

2)      Establish Configuration Environment by Configuration Manager.

3)     Baseline Configuration Items by Project Manager.

 

Establish Configuration Management Plan

The CM plan is prepared by the Configuration Manager while planning the software development activities. The configuration management plan defines:

·         Identification of Configuration Items

·         Naming convention for CIs, Storage Location of the CIs(Server/Folder name) and Baseline Criteria of CIs

·         Version Numbering of CIs

·         Baseline Labelling and Product Versioning convention (eg. 4.1)

 

 The plan is shared with all the affected groups namely Project Manager, Development Team, Analysis & Testing and their commitment obtained.

 

The CM plan evolves as the project progresses. During the project execution it may be required to add new configuration items, modify baseline criteria and modify baseline labelling or product versioning conventions. The revised CM plan should be shared with all the affected groups of the project and their commitment obtained.

 

Software Configuration Identification

 

The work products and deliverables of the project that shall be brought under configuration management depend on the type (development, maintenance) and size of project. The Configuration Controller needs to identify the CI’s. There are two groups of CI’s based on control levels:

 

A. Controlled Group

 

To control configuration item (CI’s), each must be uniquely identified. This requires the following:

·         A naming convention by which a CI can be uniquely identified.

·         A description which defines the characteristics of the CI.

·         The point at which the baseline of a CI is established.

·         The location where the CI is to be stored.

B. Managed

To manage configuration item (CI’s), each must be uniquely identified. This requires the following:

·         A naming convention by which a CI can be uniquely identified.

·         The location where the CI is to be stored.

 

To control and manage (configuration item) CI’s, each must be uniquely identified. This requires the following:

·         A naming convention by which a CI can be uniquely identified.

·         A description which defines the characteristics of the CI.

·         The point at which the baseline of a CI is established.

·         The location where the CI is to be stored.

 

Baselines

Once a baseline is created – defined – released - it can never change. If a change is made, it is a new baseline. Once a CI is approved it is checked into a configuration management system and is tagged / labeled appropriately and locked. A record of the check in is made in the CM system. Any number of changes can be made to a CI before baseline is established. Once a baseline is established for a CI, changes can be made in a controlled manner by adhering to the change control process. Changes to a baseline CI result in new baseline version of the CI. This provides for tracking the changes made to a CI and if required the CI can be rolled back to undo a change.

 

Build/Releases are tagged / labeled on meeting baseline criteria as specified in the CM plan within Project Charter. Each CI’s tag/label is related to labels of predecessors and siblings. This relationship must be made very obvious.

Establish Configuration Environment

The Configuration Manager creates a list of all the users of the CM tool for this project. An email is sent to the software project manager to authorize the creation of the users in the CM Tool. On approval from the software project manager a project folder is created in the CM Tool. The lead shall ensure that proper access rights should be given to users.

 

Create directories under the project folder. Separate directories for documents and software related products. Appropriate sub levels to be created. All files being checked in or checked out will be staged in this directory.

 

Using the list of users and their roles in the project create read/ write/ update/ delete access rights on all the directories in the CM Tool. If required create the access rights on each configurable item within the directories.

 

Perform Configuration Audit

Configuration audits shall be conducted by Configuration Manager to ensure that what has been designed has in fact been built and the tests applied to a CI prove that requirements for a CI have been satisfied. Configuration audits are required to be conducted for every phase before exiting the phase.
