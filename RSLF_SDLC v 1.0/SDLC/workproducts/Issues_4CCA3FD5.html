Description
Brief Outline

The following views should be available as minimim for viewing the current status of the work product:

<Title>

The title provides a concise overview of the Issue to be completed.

<Assign To>

The appropriate owner for the Issue.

<State>

Current state of the Issue.

<Reason>

The reason the state is in the current Issue.  

<Area Path>

The area of the application with which this Issueis associated.

<Iteration>

The iteration of the application with which this Issue is associated.

<Priority>

The level of importance for the work item on a scale of 1 (most important) to 4 (least important).

<Triage>

The level of triage for the work item that are in the Proposed state. Valid values are Pending (default), More Info, Info Received, and Triaged.

<Escalate>

Yes if the issue is affecting the critical path of the project plan.

<Details>

The Issue summary and overview.

<Analysis>

provide details about how the team can resolve the issue:

In the Impact on project promise list, the impact level:

1 - Critical, 2 - High, 3 - Medium, and 4 - Low. By default, this value is 3 - Medium.

In the Analysis, the root cause of the issue and one or more solutions that might resolve it.

<Corrective Action>

Specifies the following types of information:

In the Target resolve date, the date by which the team must resolve the issue.

In the Original Estimate, type a number that represents the hours of work that the issue will take to address.
Notation

Typical workflow progression:

- A team member creates an issue in the Proposed state with the default reason, New.

- A team member changes the state of the issue from Proposed to Active with the default reason, Accepted.

- A team member changes the state of the issue from Active to Resolved when the team has completed tasks to take corrective action.

- A team member changes the state of the issue from Resolved to Closed when the team has determined that the corrective action has addressed the problem.

 

A typical transitions:

- A team member changes the state of the issue from Proposed to Closed with the default reason, Rejected.

- A team member changes the state of the issue from Active to Proposed with the default reason, Investigation Complete.

- A team member changes the state of the issue from Active to Closed after the team determines that the issue has been overtaken by other events and is not relevant.

- A team member changes the state of the issue from Resolved to Active when the team determines that the corrective action is insufficient or incorrect for addressing the issue.

- A team member changes the state of the issue from Closed to Active after the team determines that the issue was closed in error or is reoccurring.
